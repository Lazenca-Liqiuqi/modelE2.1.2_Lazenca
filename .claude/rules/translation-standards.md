# ModelDescription文档翻译规范

## 一、文件位置规范

### 1.1 目录结构原则

翻译文件必须保持原有目录结构，不得创建新的子目录。

**规则**：
- 原始文件：`old-doc/ModelDescription/*.html`
- 翻译文件：`doc/ModelDescription/*.md`

**反例**：
```
❌ doc/technical-reference/atmosphere/
❌ doc/technical-reference/land/
```

**正例**：
```
✅ doc/ModelDescription/
```

### 1.2 文件命名

- 保持原文件名，仅改变扩展名：`.html` → `.md`
- 例如：`Atmospheric_model.html` → `Atmospheric_model.md`

## 二、术语翻译规范

### 2.1 避免口语化表达

技术术语必须使用标准、正式的中文表达，避免口语化或过于简略的译法。

| 不推荐译法 | 推荐译法 | 说明 |
|-----------|----------|------|
| 植被土 | 植被覆盖土壤 | "植被土"过于口语化 |
| 变深度层 | 可变深度层 | 加上"可"字更规范 |
| 根深 | 根系深度 | 避免过于简略 |
| 地表蓄水层 | 地表蓄水 | "层"字可能引起歧义 |

### 2.2 专有名词处理

**原则**：保留英文专名，必要时用中文说明。

| 英文术语 | 处理方式 | 示例 |
|---------|----------|------|
| TOPmodel | 保留英文 | "TOPmodel方法" |
| NASA GISS | 保留英文 | "NASA GISS ModelE" |
| MathJax | 保留英文 | "MathJax系统" |

### 2.3 新术语处理流程

遇到词典中未收录的术语时：
1. 查阅相关领域权威资料
2. 选择最标准、最通用的译法
3. 记录到 `.claude/terminology-dictionary.md` 供后续参考

## 三、句式翻译规范

### 3.1 避免"直译+括号解释"模式

**反例**：
```
蒸散发从地表蓄水层和下部（作为根深函数的）获取水分。
```

**正例**：
```
蒸散发从地表蓄水与下部土层取水，其取水深度是根系深度的函数。
```

**改进原则**：
- 将括号内的解释融入主句
- 使用"是...的函数"等自然中文表达
- 避免生硬的"作为...的"结构

### 3.2 保持句子流畅性

**检查方法**：
- 朗读译文，检查是否通顺
- 避免欧化句式
- 确保技术术语与日常用语搭配自然

**示例**：
```
❌ 可具有可变范围
✅ 其覆盖范围可变
```

## 四、格式规范

### 4.1 中英对照叠放格式

**标准格式**：
```markdown
# Title / 标题

English paragraph one.
英文段落一。

English paragraph two.
英文段落二。
```

**要求**：
- 标题：`# English / 中文`
- 段落：英文在上，中文在下
- 中英文之间空一行
- 保持原有段落结构

### 4.2 特殊内容处理

#### LaTeX公式
- 公式本身不翻译
- 说明文字需要翻译

**示例**：
```markdown
Here is an example of in-line formula: $W_i = x$
这是一个行内公式示例：$W_i = x$
```

#### 文献引用
- 保持原格式不变
- 作者姓名可音译

**示例**：
```
(Stieglitz, 1994)
(Stieglitz, 1994)
```

#### 代码块
- 代码内容不翻译
- 注释可翻译

## 六、常见问题与解决方案

### 问题1：术语词典未覆盖
**解决方案**：
1. 查阅专业文献
2. 选择标准译法
3. 更新词典文件

### 问题2：原文拼写错误
**示例**：`Evapotransiration` → `Evapotranspiration`
**处理方式**：
- 保留原文拼写（保持可追溯性）
- 或在注释中说明正确拼写

### 问题3：测试内容是否翻译
**原则**：
- 测试说明文字：翻译
- 测试代码/公式：不翻译
- 标题：翻译
